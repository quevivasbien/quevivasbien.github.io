Raytracing in JavaScript

While working on my fractal explorer, I became interested in the capabilities of code run in-browser. At the same time, I also developed some interest in computer graphics, so I decided to combine those two interests and try creating a simple ray tracing engine using JavaScript.

I found this helpful walkthrough that explains how to do pretty much exactly what I wanted. I used this tutorial as a framework for my own code, then made some changes to the details of the calculations and added a few features to make things more exciting, including the ability to move about the scene and a basic gravity simulation so the objects in the scene would orbit each other. You can try it out here.

The idea of raytracing is not especially complicated and should be implementable by anyone with some coding experience and understanding of basic vector math. I think this is the draw of raytracing in general: compared to methods like rasterization, it is easy to implement and yields realistic-looking scenes. Of course, the downside is that it is very computationally intense. As of now, only the latest-generation, top-of-the-line, graphics cards are capable of using ray tracing to render modern video games in real time. As graphics hardware gets better, I imagine that ray tracing will become the norm for rendering in video games, but up to now it has mostly been used for things like rendering animated movies or static scenes that don't need to be done in real time. All that said, I was quite pleased by how well JavaScript is able to run a simple ray-traced scene. Granted, the scenes I'm creating are very minimal, but then again, JavaScript is not particularly fast, and this is all running on a single CPU thread.

Wanting to experiment a bit more with WebAssembly compilation, I proceeded to make a version of the ray tracing engine in C++, which you can see here. Unfortunately, this actually runs slower than the pure JavaScript version (at least on Chrome; I got different results on Firefox). I think that is probably due to some C++ feature I used that is not well-supported by Emscripten, or just because executing things in WebAssembly requires a bit of a fixed cost per call that becomes cumbersome when you're trying to push out many frames per second. If the latter scenario is the case, then I'd need to shift more of the code pipeline to C++ so there is less passing of memory between JavaScript and WebAssembly/C++. I may go back and work on optimizing things, but for now I'm satisfied with watching my little simulated planets in motion, enjoying their lambert shading and specular reflections.
